RetroTector

 

RetroTector

General description

RetroTector is a program package designed to identify and analyze possible endogenous retroviruses in DNA sequences. It is highly flexible, but is in its present form somewhat slanted towards primate genomes.

RetroTector has four main functions:
  • To search for possible Long Terminal Repeats, both paired and single (performed by the LTRID module).
  • To search for evidence of endogenous retroviruses, or possibly other elements, in DNA sequences, and characterize the findings (mainly performed by the RetroVID module).
  • To construct putative proteins ("puteins") similar to those originally encoded by the virus (mainly performed by the ORFID module).
  • To identify other possible exons (performed by the XonID module).
Besides these main modules, there are a number of others for presentation of the results, storage in an SQL database etc.

Files in RetroTector

The RetroTector package includes a considerable number of files, which may be subdivided thus:
Java Bytecode files. These have been generated by the Java compiler. They consist of Java classes, which are organized in two ways:
  • .jar files. These contain a number of classes, in a format related to .zip. The central module, RetroTector.jar (or possibly RetroTector|version number|.jar)is of this type.
  • .class files. These contain one class each. If you are a Java programmer, you can make your own ones. They are not necessary but constitute a convenient way of adding modules. They are contained in the 'plugins'  directory.
Text files. These have the .txt extension. Most of them conform to the Parameter file format. You may view them using any text editing program and if you feel that you know what you are doing, you may alter them yourself. Just make sure that you save them in simple text format, not for example Word format. There is also a rudimentary text editing facility in RetroTector.

forretrotector.sql. This is an SQL script which, when presented to an SQL database manager (preferably MySQL) generates tables used by RetroTector.

Directory organization

The main directory may be named anything, but here it will be assumed to have the name
Retrotector
It must contain
RetroTector.jar
(though you may change its name, as long as you change the start command accordingly) and a directory named
Database
and may also contain a directory named

plugins

It is recommended that you collect your work in a subdirectory of the Retrotector directory. It may have any name, but we will assume

Workplace
It in turn should have a subdirectory named
NewDNA

This is where you put your new DNA files (see SweepDNA about file formats), whereupon the SweepDNA command will distribute them in individual subdirectories of Workplace. After that, the SweepScripts command will typically be invoked, to process them automatically.

Running RetroTector

RetroTector is written in strict Java 2, and should therefore run on any computer equipped with a Java Runtime Environment. For all features to operate, Java 1.4.2 or later is recommended.

The start command

The command to actually start RetroTector is implementation dependent and usually rather complex. Before it is applied, the current directory should be set to Retrotector.  The command will have to specify
 
Class path. This tells the Java Runtime Environment where to look for classes. It should specify
  • RetroTector.jar
  • The path to the SQL Java driver, if you use SQLUser or any of its subclasses.
  • The path to the class files in the 'plugins' directory, if present. Normally this will be the Retrotector directory.
Starting class. This is 'retrotector/RetroTector'. Be careful with character case!
There may also be a need to specify
Optional parameters for RetroTector. These may be (and will be handled in consecutive order when RetroTector starts up):
  • 'L:' followed by the first part of a file name. Provided it is the first parameter, it will be extended with a version number and '.txt' to form the name of the log file, instead of LatestErrorLog.
  • The name of an executor, e g 'SweepDNA'. It will be executed.
  • 'D:' followed by the full name of a directory. It will be made the working directory.
  • 'E:' followed by the name of an executor script. It will be executed.
  • 'F:' followed by the name of a file, from which parameters according to this list will be fetched.
  • 'Quit'
Implementation dependent parameters. For example, the amount of memory to occupy may have to be specified. We tend to use 768 MB (though twice as much is nice if you have it). Thus, a likely command line to start RetroTector under Sun Java for Windows could be (one line, though it may not look so here)

java -mx768m -cp C:\Retrotector\RetroTector.jar;C:\Retrotector\sqldriver.jar;C:\Retrotector retrotector/RetroTector

Under MacOS X (ie Unix)

java -Xmx768m -classpath "RetroTector.jar:sqldriver.jar:." retrotector/RetroTector
(again on a single line) is a possibility.
Please note that if you specify too much memory relative to the actual RAM in you computer (for example 768 MB on a 512 MB computer) much of the work will still go smoothly, but some operations (in particular ORFID and CollectGenome) may resort to unsettling and very time-consuming swapping. On the other hand, if you specify less, you may get OutOfMemoryErrors instead. In the end, you may just have to get more RAM.
When RetroTector starts, the RetroTector window appears, and you carry on from there.

At first startup, a Notice, 'Properties file could not be read' may appear. Don't worry; the Properties file (Properties.txt) will be generated automatically.

Though it is platform-dependent, in general it is possible to have more than one instance of RetroTector running simultaneously, but the above warning about sufficient RAM applies even more.

    Control via standard input

It is also possible to give commands to RetroTector at any time, via the standard input. The commands should have the same syntax as the 'Optional parameters' above. Normally, the standard input is the text window, but under a suitable operating system, it may be reassigned, offering the possibility of control of RetroTector by another program, via a "pipeline".


  .zip files and RetroTector

The output from a major RetroTector job may stretch to gigabytes. You may reduce that by about 70% by compressing it into one or more .zip files. In most situations where text files are to be accessed read-only, RetroTector can reach inside a .zip file and use its contents without actually decompressing it.
.zip files will then appear in the file choice dialogs as roughly equivalent to directories.

This function depends on the Java package java.util.zip. At this date, it seems that it cannot handle all .zip files, in particular not files zipped by Windows. WinZip works, however, at least as long as only one computer is involved.


RetroTector and computer clusters

If available, it may be advantageous to run big RetroTector jobs on multiple processors, eg one processor per chromosome. However, the finishing work by CollectGenome cannot be conveniently distributed. It may then be advantageous to add a line

ClusterMode: Yes

to Configuration.txt. This will cause RetroVID and ORFID to take over some tasks normally performed by CollectGenome.

For this kind of work, a stripped version, RetroTectorEngine, is available. It has no user interface and does not require any Java graphics.
